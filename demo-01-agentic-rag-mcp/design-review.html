§00

The honest headline

This is not a throughput system. When you run the scale math in §01, average load lands around 0.03 conversations/second, peaking near 0.3/s on a sale day. A single application server handles that without strain. Walking into a review talking about sharding, autoscaling, and QPS would be solving a problem this system does not have.

The problem it actually has is correctness and blast radius. It is an autonomous agent that moves money — it issues refunds via Razorpay — and makes factual policy claims to customers. Every failure worth designing against in Demo 01's nine-scenario matrix was a system-design failure, not a model-capability failure: silent retrieval misses, parallel tool calls widening blast radius, an async webhook gap that let the agent report a refund as done before it was confirmed, missing idempotency, and policy contradictions surfaced verbatim to users.

Thesis

The rate of money-moving actions and the cost of a wrong autonomous action are the real scale axes here — not requests per second. Every section below is organized around that.

§01

Requirements & scale assumptions

Functional requirements

#RequirementNotes
F1Answer customer policy questions (returns, refunds, shipping, cancellation windows) grounded in a curated corpusRAG, not parametric knowledge
F2Take bounded actions: check order status, initiate refund / returnSide-effecting · money movement
F3Escalate to a human when confidence is low or the request is out of scopeThe default, not the exception
F4Report action status truthfully, including “in progress / pending confirmation”Directly from Scenario 9

Non-functional requirements — ranked by what actually matters

  1. Correctness of money-moving actions — no double refunds, no over-refunds, no ineligible refunds. The top constraint.
  2. Truthful state reporting — never claim an action succeeded before it is confirmed.
  3. Grounded answers — never answer a policy question from parametric memory when retrieval returns nothing relevant.
  4. Graceful degradation — when a tool or the payment provider fails, escalate cleanly rather than improvise.
  5. Latency — a few seconds end-to-end is fine for support; not latency-critical.
  6. Throughput — effectively a non-requirement at this scale (proven below).

Peak load is ~1 QPS of model calls. Throughput is not a design problem. The number worth obsessing over is ~290 money-moving actions/day — because the cost of getting one wrong dwarfs everything else in this system.

Open question — flag, don't paper over

The 7% contact rate and 50% addressable share are typical-but-assumed. If real ticket data shows the addressable share is far lower (mostly out-of-scope disputes), the agent's ROI weakens and a narrower scope may be the right call. Validate against the actual support-ticket taxonomy before committing.

§03

Component design

3.1 — Host / Orchestrator (host.py)

What it does: runs the explicit Anthropic tool-use loop — sends the user message plus tool definitions to Claude, executes returned tool calls, feeds results back, repeats until a final answer.

Why not a framework: the explicit loop was chosen over LangGraph / CrewAI / LlamaIndex deliberately. For a system whose dominant risk is blast radius of autonomous actions, framework opacity is a liability — you need to see and gate every tool call. The trade-off is real: you write orchestration plumbing yourself. At this scale that's the right trade; the plumbing is small and the visibility is worth more than the saved code.

Where it breaks (observed): parallel tool calls increase blast radius. When the model emits multiple tool calls in one turn, side-effecting ones can fire alongside reads, widening the surface for a wrong action.

Mitigation: partition tools into read and side-effecting. Reads parallelize freely. Side-effecting tools are serialized and gated — at most one in flight, each preceded by a server-side eligibility check (§05). This is the single most important control in the system.

3.2 — Retrieval layer (rag.py)

What it does: embeds the query with Voyage AI, retrieves top-k chunks from the policy corpus, hands them to the host as grounding context.

Where it breaks (observed): silent retrieval misses — the query embeds, retrieval returns something (nearest neighbours always exist), but nothing is relevant, and the agent answers anyway from parametric knowledge. The most dangerous RAG failure because it is invisible.

Mitigation: a retrieval-confidence threshold. If the top result's similarity is below a tuned floor, treat it as “no relevant policy found” → the agent must say it doesn't know and escalate, never improvise. Tune the floor against labelled in-corpus and out-of-corpus queries; accept that this trades some auto-resolution for correctness. For a system making policy claims to customers, that trade is correct.

Honest scale note — the vector store

The corpus is ~12 docs, growing to maybe a few hundred. This does not justify a managed vector DB. In-memory or pgvector on the existing Postgres suffices and removes an entire piece of infrastructure. Reach for Pinecone/Qdrant only past ~10⁵–10⁶ chunks or for filtered hybrid search at scale. A vector DB here would be resume-driven architecture, not problem-driven.

3.3 — MCP Server (mcp_server.py) and the three tools

Exposes get_order_status, search_policy, issue_refund. The first two are reads. issue_refund is the money-moving tool and gets the eligibility gate, idempotency key, and serialization described above and in §05.

3.4 — Payments path and the webhook reconciliation gap

The observed failure (Scenario 9): the agent calls issue_refund → Razorpay returns “initiated” → the confirmed status arrives asynchronously via webhook 4–5 seconds later. In between, a naive implementation would report “refund successful” before it actually is.

Mitigation — never conflate “initiated” with “confirmed”:

Same class of bug as the silent-webhook observability defect already caught in the Razorpay lab — designed out here rather than patched after the fact.

§05

Threat model — money movement is the domain risk

Generic LLM-security checklists are weak here. The specific risk is precise: an autonomous agent that can move money, driven by untrusted user text.

ThreatVectorControl
Prompt injection → unauthorized refundUser message instructs the agent to refundEligibility + amount decided server-side from the order record, never from agent/user values. The agent requests; the payments service authorizes.
Over-refund / double-refundRepeated or inflated refund callIdempotency keys; server-side cap at original paid amount; one-refund-per-order-line invariant
Refund to ineligible orderAgent misreads policyEligibility check in payments service is the source of truth, independent of the agent's reasoning
Policy corpus poisoningTampered doc → wrong customer-facing policyCorpus write access controlled + reviewed; treat the corpus as production config
PII leakageOrder/customer data in prompts and logsRedact PII from logs; minimize PII in retrieved context; scoped per-conversation access
The load-bearing principle

The agent is never the authority on whether or how much to refund. It proposes; a deterministic server-side check disposes. This makes prompt injection an annoyance rather than a financial incident.

The honest cost story: at this scale inference is ~$2.6K/month — and that is not where cost concentrates. The real cost is human escalation labour (~460 escalations/day × handling time) and refund risk (one wrong autonomous refund can cost more than a month of inference, plus trust damage). Optimizing token cost here would be premature optimization of the cheap thing.

Cost levers, in priority order — but most are premature

  1. Reduce escalation rate via better retrieval / containment — biggest real lever; attacks the dominant cost.
  2. Route simple classification to Haiku, reserve Sonnet for synthesis — would cut inference materially, but inference isn't the bottleneck, so this is a “later.”
  3. Cache embeddings for repeated queries — marginal at this volume.

Do #1. Defer #2 and #3 until there's a reason.

§07

Monitoring & observability

Business health

Auto-resolution (containment) rate · escalation rate · refund volume & value/day · CSAT on agent-handled conversations.

System health

Tool-call latency & error rate · retrieval latency · LLM error / rate-limit rate · payments-API error rate.

Correctness — the ones that matter most, and the hardest to measure

  • Retrieval-miss rate (queries below confidence floor)
  • Refund reconciliation lag (INITIATED → CONFIRMED time; the Scenario-9 metric)
  • Stuck-refund count (INITIATED beyond SLA → poller should have caught it)
  • Policy-contradiction flags raised
Open question — unsolved, not hidden

Hallucinated-policy-answer rate is genuinely hard to measure automatically. Candidate approach: sampled human review + an LLM-judge cross-check against retrieved chunks. Flagged as unsolved rather than pretending a dashboard covers it.

§09

What this design deliberately does not do

  • No horizontal scaling story. Proven unnecessary in §01. Adding it would solve an imaginary problem.
  • No managed vector DB. Corpus size doesn't justify it.
  • No multi-region / HA architecture. Out of scope at this scale; revisit at 20× volume.
  • No fine-tuning. Retrieval + a good base model covers the requirement; fine-tuning would be premature.

If the scale assumptions in §01 turn out 10–20× larger after validation, what changes is the human-escalation operating model and possibly the datastore — not the agent core. The agent core is correctness-bound, not throughput-bound, and that doesn't change with volume.
